Threads::FunctionCall: Functions as First-Class Objects

In many places throughout the Vrui toolkit, unrelated modules communicate with each other via events and callbacks. For example, a device driver might send a device event such as a button press to a device object, and some other module may register a callback to be notified when a button on some device object changes state.

The Vrui toolkit provides two main mechanisms of implementing event notifications between unrelated modules: the Misc::CallbackList class from the Misc library, and the Threads::FunctionCall class from the Threads library. While the older Misc::CallbackList class and its associated Misc::CallbackData base class are somewhat limited in functionality, intended for broadcast-style events, and primarily used in Vrui's UI layers, such as for GUI events in the GLMotif library or raw input device events in Vrui itself, Threads::FunctionCall is a more powerful and general mechanism that is used for all kinds of one-to-one communication purposes throughout the Vrui toolkit. Unlike Misc::CallbackList, Threads::FunctionCall approximates the ideal of functions as first-class objects that can be passed around as data, shared between threads, stored in arbitrary data structures, and that will clean up after themselves when no longer needed.

For example, while an event such as "the application's window was resized" would typically be handled via Misc::CallbackList because multiple independent clients may be interested in being notified of the same event, Threads::FunctionCall would be used for "private" events that can only be relevant to a single client, such as registering an I/O watcher for a file owned by a client with a Threads::RunLoop object, or receiving a completion notification for an asynchronous background job submitted to the Threads::WorkerPool class.

The Threads::FunctionCall Class

Threads::FunctionCall is the base class for function call objects. It is templatized by the single parameter that is passed to the functions that it represents: 

namespace Threads {
template <class ParameterParam>
class FunctionCall;
}
Threads::FunctionCall is a functor class in the sense that it declares two virtual function call operators that both return void and take a single argument of the template parameter type, but one of them is declared const and the other one is not: 
virtual void operator()(ParameterParam parameter) const;
virtual void operator()(ParameterParam parameter);
In other words, any object derived from Threads::FunctionCall<ParameterParam> can be called just like a C/C++ function with the signature 
void function(ParameterParam parameter);

Note that ParameterParam is the direct, call-by-value type of the argument of the represented C/C++ function. In other words, if a function expects a pointer or a reference to an object, such as 

void function(const SomeClass& someObject);
then ParameterParam must be const SomeClass&. In practical use, ParameterParam is typically a reference to a compound type that provides detailed information about the event that caused the function to be called.

Furthermore, Threads::FunctionCall is derived from Threads::RefCounted, meaning that the lifetimes of its objects can be managed in a thread-safe way via reference-counting smart pointers such as Misc::Autopointer.

The full declaration of Threads::FunctionCall is as follows: 

namespace Threads {

template <class ParameterParam>
class FunctionCall:public RefCounted
	{
	/* Embedded classes: */
	public:
	typedef ParameterParam Parameter; // Function call parameter type
	
	/* Constructors and destructors: */
	public:
	virtual ~FunctionCall(void);
	
	/* Methods: */
	virtual void operator()(Parameter parameter) const; // Function call operator when function object is non-modifiable
	virtual void operator()(Parameter parameter); // Function call operator when function object is modifiable
	};

}

Using Threads::FunctionCall as a Client

There are two main ways for clients to use Threads::FunctionCall: a) create a wrapper for an existing function or class method, or b) derive a custom functor class. Which one is "better" depends very much on specific circumstances and, to a degree, on the personal taste of the developer. In general, the wrapper method may be better if the function to be called does not require much additional state beyond what is provided by the caller, or is a method of a class where the containing object is directly involved in handling the event, or if the developer prefers a "fire and forget" way of registering callbacks. The functor method, on the other hand, may be better for callbacks that require a lot of additional state or have separate processing stages, are not deeply embedded with existing classes, or need to be accessed by multiple modules themselves.

Registering Callbacks

Classes that emit events typically have methods to register callbacks for said events, or take a callback object as an argument to their class constructor. In either case, the typical pattern to register a callback is to do so with a non-const reference to an object of a class derived from the template version of Threads::FunctionCall that reflects the emitted event's event data type. The reason for expecting a reference to an event object instead of a pointer is to minimize potential mistakes when managing the lifetimes of callback objects. Concretely, the procedure goes like this: 

/* Create a new callback object on the heap, following one of the approaches from the following sections: */
Threads::FunctionCall<CallbackParameter>* callback=...;

/* Pass the callback object to the registration method: */
eventSource.registerCallback(*callback); // Note: de-reference the callback object pointer here!

/* Do whatever with the callback object / pointer: */
...
Note that the callback object created in the first step is a reference-counted object with an initial reference count of zero. Meaning, if the object needs to be held any extended time beyond its initial creation and registration with an event source, it must be wrapped in a smart pointer such as Misc::Autopointer, for example: 
/* Create a new callback object on the heap, following one of the approaches from the following sections: */
Misc::Autopointer<Threads::FunctionCall<CallbackParameter> > callback=...;
This is a lot of typing for the simple task of creating a callback, so please read on to find out why it is hardly ever necessary.

In most scenarios, the client will have no further use for the freshly registered callback object in a "fire-and-forget" scenario. Maybe the callback object will be destroyed when the callback emitter itself is destroyed, or maybe there is another method that will cancel the event for which the callback was registered, in which case the callback object is destroyed at that time. Either way, the typical client-side pattern for this, assuming for simplicity that the client uses a derived functor object, is: 

eventSource.registerCallback(*new CustomCallback); // Note: de-reference the result from new here!
The fact that Threads::FunctionCall is derived from Threads::RefCounted, and that the event source will hold the passed callback object via a Misc::Autopointer, guarantees that the newly-created callback object will be destroyed at the appropriate moment and not leak memory or other held resources past its expected lifetime. Not that this call pattern makes it unnecessary to wrap the pointer to the newly-created callback object in a smart pointer. The pointer only exists between the time the object is created, and then immediately registered with the event source.

Creating Function Wrappers

The Threads/FunctionCalls.h header file provides several pre-made classes derived from Threads::FunctionCall to wrap several types of existing functions, such as global functions with or without additional arguments; class methods with or without additional arguments, functions taking const or non-const references, etc. The header also provides a templatized namespace-global helper function 

namespace Threads {
template <class ParameterParam>
FunctionCall<ParameterParam>* createFunctionCall(...);
}
that simplifies the process of creating appropriate wrappers for existing functions or methods.

For example, consider an event source class EventSource that emits events carrying information as const references to objects of class EventData. In the simplest case, the client has a namespace-global function, or a static method, that takes an argument of type const EventData&: 

void eventCallback(const EventData& eventData)
	{
	/* Do something with the EventData object... */
	}
In this case, the client would create a wrapper for that callback function via 
Threads::FunctionCall<const EventData&>* callback=Threads::createFunctionCall(eventCallback);
or, typically, directly pass the newly-created wrapper to the appropriate registration method via 
eventSource.registerCallback(*Threads::createFunctionCall(eventCallback)); // Note: de-reference the callback pointer here!
The template arguments for the Threads::createFunctionCall function are determined automatically by the compiler.

If the callback is a non-static class method, as in 

class Client
	{
	void eventCallback(const EventData& eventData);
	};
then the wrapper would instead be created thus, assuming that it is created from inside another method of class Client: 
Threads::FunctionCall<const EventData&>* callback=Threads::createFunctionCall(this,&Client::eventCallback);
