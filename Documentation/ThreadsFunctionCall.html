Threads::FunctionCall: Functions as First-Class Objects

In many places throughout the Vrui toolkit, unrelated modules communicate with each other via events and callbacks. For example, a device driver might send a device event such as a button press to a device object, and some other module may register a callback to be notified when a button on some device object changes state.

The Vrui toolkit provides two main mechanisms of implementing event notifications between unrelated modules: the Misc::CallbackList class from the Misc library, and the Threads::FunctionCall class from the Threads library. While the older Misc::CallbackList class and its associated Misc::CallbackData base class are somewhat limited in functionality, intended for broadcast-style events, and primarily used in Vrui's UI layers, such as for GUI events in the GLMotif library or raw input device events in Vrui itself, Threads::FunctionCall is a more powerful and general mechanism that is used for all kinds of one-to-one communication purposes throughout the Vrui toolkit. Unlike Misc::CallbackList, Threads::FunctionCall approximates the ideal of functions as first-class objects that can be passed around as data, shared between threads, stored in arbitrary data structures, and that will clean up after themselves when no longer needed.

For example, while an event such as "the application's window was resized" would typically be handled via Misc::CallbackList because multiple independent clients may be interested in being notified of the same event, Threads::FunctionCall would be used for "private" events that can only be relevant to a single client, such as registering an I/O watcher for a file owned by a client with a Threads::RunLoop object, or receiving a completion notification for an asynchronous background job submitted to the Threads::WorkerPool class.

The Threads::FunctionCall Class

Threads::FunctionCall is the base class for function call objects. It is templatized by the type of the single parameter that is passed to the functions that it represents: 

namespace Threads {
template <class ParameterParam>
class FunctionCall;
}
Threads::FunctionCall is a functor class in the sense that it declares two virtual function call operators that both return void and take a single argument of the template parameter type, but one of them is declared const and the other one is not: 
virtual void operator()(ParameterParam parameter) const;
virtual void operator()(ParameterParam parameter);
In other words, any object derived from Threads::FunctionCall<ParameterParam> can be called just like a C/C++ function with the signature 
void function(ParameterParam parameter);

Note that ParameterParam is the direct, call-by-value type of the argument of the represented C/C++ function. In other words, if a function expects a pointer or a reference to an object, then the template parameter must include that reference or pointer. For example, if a function is declared as 

void function(const SomeClass& someObject);
then ParameterParam must be const SomeClass&. In practical use, ParameterParam is typically a reference to a compound type that provides detailed information about the event that caused the function to be called.

Furthermore, Threads::FunctionCall is derived from Threads::RefCounted, meaning that the lifetimes of its objects can be managed in a thread-safe way via reference-counting smart pointers such as Misc::Autopointer.

The full declaration of Threads::FunctionCall is as follows: 

namespace Threads {

template <class ParameterParam>
class FunctionCall:public RefCounted
	{
	/* Embedded classes: */
	public:
	typedef ParameterParam Parameter; // Function call parameter type
	
	/* Constructors and destructors: */
	virtual ~FunctionCall(void);
	
	/* Methods: */
	virtual void operator()(Parameter parameter) const; // Function call operator when function object is non-modifiable
	virtual void operator()(Parameter parameter); // Function call operator when function object is modifiable
	};

}

Using Threads::FunctionCall as a Client

There are two main ways for clients to use Threads::FunctionCall: a) create a wrapper for an existing function or class method, or b) derive a custom functor class. Which one is "better" depends very much on specific circumstances and, to a degree, on the personal taste of the developer. In general, the wrapper method may be better if the function to be called does not require much additional state beyond what is provided by the caller, or is a method of a class where the containing object is directly involved in handling the event, or if the developer prefers a "fire and forget" way of registering callbacks. The functor method, on the other hand, may be better for callbacks that require a lot of additional state or have separate processing stages, are not deeply embedded with existing classes, or need to be accessed by multiple modules themselves.

Registering Callbacks

Classes that emit events typically have methods to register callbacks for said events, or take a callback object as an argument to their class constructor. In either case, the typical pattern to register a callback is to do so with a non-const reference to an object of a class derived from the template version of Threads::FunctionCall that reflects the emitted event's event data type. The reason for registration methods expecting a reference to a function object instead of a pointer is to minimize potential mistakes when managing the lifetimes of function objects. Concretely, the procedure goes like this: 

/* Create a new callback object on the heap, following wrapper or functor approaches described in the following sections: */
Threads::FunctionCall<CallbackParameter>* callback=...;

/* Pass the callback object to the registration method: */
eventSource.registerCallback(*callback); // Note: de-reference the callback object pointer here!

/* Do whatever with the callback object / pointer: */
...
Note that the callback object created in the first step is a reference-counted object with an initial reference count of zero. Meaning, if the object needs to be held any period of time beyond its initial creation and registration with an event source, it must be held by a smart pointer such as Misc::Autopointer, for example: 
/* Create a new callback object on the heap, following one of the approaches from the following sections: */
Misc::Autopointer<Threads::FunctionCall<CallbackParameter> > callback=...;
This is a lot of typing for the simple task of creating a callback, so please read on to find out why it is hardly ever necessary.

In most scenarios, the client will have no further use for the freshly registered callback object; i.e., the most common use case is a "fire-and-forget" scenario where maybe the callback object will be destroyed when the event-emitting object itself is destroyed, or maybe the event-emitting object has another method that will cancel the event for which the callback was registered, in which case the callback object is destroyed by the emitting object at that time. Either way, this is implemented by the event source object holding the callback function with a reference-counting smart pointer. The typical client-side pattern for this use case, assuming for simplicity that the client uses a derived functor object, is: 

eventSource.registerCallback(*new CallbackFunctor); // Note: de-reference the result from new here!
The facts that Threads::FunctionCall is derived from Threads::RefCounted, and that the event source will hold the passed callback object via a Misc::Autopointer, guarantee that the newly-created callback object will be destroyed at the appropriate moment and not leak memory or other held resources past its expected lifetime. Note that this call pattern makes it unnecessary to wrap the pointer to the newly-created callback object in a smart pointer: the pointer only exists between the time the object is created, and then immediately registered with the event source, where it will be held with a smart pointer.

Creating Function Wrappers

The Threads/FunctionCalls.h header file provides several pre-made classes derived from Threads::FunctionCall to wrap several types of existing functions, such as global functions with or without additional arguments, class methods with or without additional arguments, functions taking const or non-const references, etc. The header also provides a templatized namespace-global helper function 

namespace Threads {
template <class ParameterParam>
FunctionCall<ParameterParam>* createFunctionCall(...);
}
that simplifies the process of creating appropriate wrappers for existing functions or methods.

For example, consider an event source class EventSource that emits events carrying information as const references to objects of class EventData. In the simplest case, the client has a namespace-global function, or a static method, that takes an argument of type const EventData&: 

void eventCallback(const EventData& eventData)
	{
	/* Do something with the EventData object... */
	}
In this case, the client would create a wrapper for that eventCallback function via 
Threads::FunctionCall<const EventData&>* callback=Threads::createFunctionCall(eventCallback);
or, typically, directly pass the newly-created wrapper to the appropriate registration method via 
eventSource.registerCallback(*Threads::createFunctionCall(eventCallback)); // Note: de-reference the result pointer here!
The template arguments for the Threads::createFunctionCall function are determined automatically by the compiler.

If the callback is a non-static class method, as in 

class Client
	{
	void eventCallback(const EventData& eventData);
	};
then the wrapper would instead be created thus, assuming that it is created from inside another method of class Client: 
Threads::FunctionCall<const EventData&>* callback=Threads::createFunctionCall(this,&Client::eventCallback);
or 
eventSource.registerCallback(*Threads::createFunctionCall(this,&Client::eventCallback)); // Note: de-reference the result pointer here!

Passing Additional Arguments to Wrapped Functions

In many cases, the per-event data provided by a callback's main parameter is not sufficient to handle the callback. Callbacks often have additional state associated with them that is outside the purview of the event source and has to be communicated separately. For this common use case, the Threads/FunctionCalls.h header file defines a category of wrappers for functions or non-static class methods that take one additional, event-independent, argument of an arbitrary type. This is the analog of the C-style callback pattern where callback functions receive an opaque void* userData pointer. Concretely, this might look like this: 

void eventCallback(const EventData& eventData,int eventId)
	{
	/* The eventId will always be 42 in this example, regardless of the event or content of eventData: */
	std::cout<<"Event callback with ID "<<eventId<<std::endl;
	}

...

/* Register a callback that receives a fixed additional argument of type int: */
eventSource.registerCallback(*Threads::createFunctionCall(eventCallback,42)); // Note: de-reference the result pointer here!
Importantly, the additional argument is passed by value to the callback function. Meaning, if the argument type is a compound and potentially complex data type and should be passed by reference or pointer, then the additional argument must be declared as a reference or a pointer, and passed to the Threads::createFunctionCall function in a compatible type. Specifically, the Threads::createFunctionCall takes the additional argument as a const reference, unless the argument is of a non-const reference type in the callback function, in which case the argument is expected in that same type.

In detail, the type correspondences are as follows:

  • Callback function takes the additional argument as const ArgumentParam&: Threads::createFunctionCall takes the additional argument as const ArgumentParam& and passes it to the callback function directly as that.
  • Callback function takes the additional argument as ArgumentParam&: Threads::createFunctionCall takes the additional argument as ArgumentParam& and passes it to the callback function directly as that.
  • Callback function takes the additional argument as a const pointer const ArgumentParam*: Threads::createFunctionCall takes the additional argument as a const reference to const ArgumentParam* and passes it to the callback function by value as const ArgumentParam*.
  • Callback function takes the additional argument as a non-const pointer ArgumentParam*: Threads::createFunctionCall takes the additional argument as a const reference to ArgumentParam* and passes it to the callback function by value as ArgumentParam*.
  • Callback function takes the additional argument as anything else: Threads::createFunctionCall takes the additional argument as a const reference to ArgumentParam, i.e., as const ArgumentParam&, stores it internally as a copy of type ArgumentParam, and passes that copy to the callback function by value as ArgumentParam.
In practice, this "just works," and the client developer does not have to worry about the exact details of how it works.

Managing the Lifetime of Additional Function Arguments

If a callback function requires a complex data structure as additional argument that cannot, or should not, be copied inside the callback wrapper and passed to the callback function by value, and therefore needs to be passed as a reference or pointer, it raises the question of who is responsible for disposing of that additional argument once it is no longer needed. In C-style callbacks, outside of simple use cases, this either leads to excessively complicated code managing the lifetimes of all those per-callback structures, or to memory and other resource leaks. With Threads::FunctionCall, this problem is elegantly solved by deriving reference-passed additional argument data structures from the Threads::RefCounted base class, and passing them to callback functions as smart pointers. Concretely, 

class AdditionalArgument:public Threads::RefCounted
	{
	...
	};

typedef Misc::Autopointer<AdditionalArgument> AdditionalArgumentPointer; // Type for smart pointers to AdditionalArgument objects

void eventCallback(const EventData& eventData,AdditionalArgumentPointer argument) // Note: the smart pointer is passed by value
	{
	...
	}

...

eventSource.registerCallback(*Threads::createFunctionCall(eventCallback,AdditionalArgumenPtr(new AdditionalArgument))); // Note: wrap the additional argument in a smart pointer and de-reference the result pointer here!
Note that, as always, the additional argument is passed by value, in this case as a value of type Misc::Autopointer, and that the additional argument must be explicitly wrapped in a Misc::Autopointer when passed to Threads::createFunctionCall.

Some future version of the Threads library will have an additional category of wrapper classes where Threads::createFunctionCall will accept a naked pointer, such as from new ArgumentParam, hold it internally using a smart pointer, and then pass it to the callback function by reference as an ArgumentParam&. Stay tuned.

In general, if the additional argument structures become more complicated to define and to manage, it might be time to switch to handling callbacks by creating custom functor classes, which have none of those issues.

Creating Functor Classes

Implementing callbacks as functor classes is straightforward. One simply has to declare a new class derived from the appropriate Threads::FunctionCall template, declare all state necessary to handle callbacks as elements, and finally implement the const and/or non-const version of the virtual operator()(ParameterParam parameter) method: 

class EventCallback:public Threads::FunctionCall<const EventData&>
	{
	/* Elements: */
	private:
	... // All state that is needed to handle callbacks...
	
	/* Constructors and destructors: */
	public:
	EventCallback(...) // Takes all state that is needed to later handle callbacks
	virtual ~EventCallback(void); // Releases all resources held by the callback object
	
	/* Methods from class Threads::FunctionCall<const EventData&>: */
	virtual void operator()(const EventData& eventData) const
		{
		/* Handle the callback; cannot modify the internal state of the callback object: */
		...
		}
	virtual void operator()(const EventData& eventData)
		{
		/* Handle the callback; can modify the internal state of the callback object: */
		...
		}
	};

...

/* Register a new callback, passing necessary state to the EventCallback constructor: */
eventSource.registerCallback(*new EventCallback(...)); // Note: de-reference the result from new here!

As Threads::FunctionCall is itself derived from Threads::RefCounted, the lifetimes of such function objects will be managed automatically.

In the common case where function objects need to interact closely with objects of some class, the functor class is typically declared as a sub-class of that class, declared friend, and holds as part of its state a reference to an object of the parent class, which is passed as an argument to its constructor: 

class EventSink
	{
	/* Embedded classes: */
	private:
	class EventCallback:public Threads::FunctionCall<const EventData&>
		{
		/* Elements: */
		private:
		EventSink& eventSink; // Reference to the object to which this callback belongs
		... // All state that is needed to handle callbacks...
		
		/* Constructors and destructors: */
		public:
		EventCallback(EventSink& sEventSink,...) // Takes a reference to the owning object and all state that is needed to later handle callbacks
			:eventSink(sEventSink),...
			{
			}
		virtual ~EventCallback(void); // Releases all resources held by the callback object
		
		/* Methods from class Threads::FunctionCall<const EventData&>: */
		virtual void operator()(const EventData& eventData) const
			{
			/* Handle the callback; cannot modify the internal state of the callback object: */
			...
			}
		virtual void operator()(const EventData& eventData)
			{
			/* Handle the callback; can modify the internal state of the callback object: */
			...
			}
		};
	
	friend class EventCallback; // Declare the functor class as a friend
	
	...
	
	void registerForEvent(EventSource& eventSource) // Register a callback with the given event source
		{
		/* Register a new callback, passing a reference to this object and necessary state to the EventCallback constructor: */
		eventSource.registerCallback(*new EventCallback(*this,...)); // Note: de-reference the result from new here!
		}
	};
